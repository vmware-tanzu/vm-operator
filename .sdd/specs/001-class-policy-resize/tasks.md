This was deliberately **not** unified further into one shared "evaluate against a VM" entry point that both callers invoke identically end-to-end: the webhook must attach a specific `field.Path` to each violation (and, for `vmClassMode`/`createMode`/`updateMode`/`powerOnMode` gating, must decide per-mode whether to block admission at all), while the provider's condition logic needs to distinguish "not subject to any policy" from "subject and compliant" from "subject and non-compliant" independent of `powerOnMode` ‚Äî decisions that depend on caller-specific context `Validate` intentionally has no knowledge of. `configpolicy.AppliesToVM(spec, className)` centralizes only the `vmClassMode` bypass check (shared, but kept outside `Validate` so callers can still branch on it separately, e.g. to delete vs. evaluate the condition). Webhook translation of violations to field paths lives in `toFieldErrors`/`extraConfigFieldError` in `virtualmachine_validator_configpolicy.go` (the `extraConfig` case re-locates the violating key's slice index, since `Input` only carries keys, not indices). Tests: `pkg/util/configpolicy/configpolicy_test.go` (62 specs ‚Äî every violation type, `Input` builders, `AppliesToVM`, `Violations` flattening including nested joins); webhook unit tests extended in `virtualmachine_validator_configpolicy_test.go`; provider unit tests extended in `vmprovider_vm_power_test.go` (adherent/non-adherent/no-policy/unplaced/VM-Class-bypass cases for the new condition).
